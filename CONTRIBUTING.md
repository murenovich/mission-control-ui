# Contributing

This repo uses a lightweight workflow optimized for clean pull requests, predictable versioning, and low-friction releases.

## Branches

- branch from `main`
- use a short descriptive name
- prefer the `codex/<description>` pattern for feature work

Examples:

- `codex/newsfeed-filters`
- `codex/fix-sidebar-scroll`
- `codex/release-workflow`

## Pull Requests

1. Create a focused branch from `main`.
2. Make the smallest complete change that solves the problem.
3. Run `npm run ci` locally before opening the PR.
4. Fill out the PR template with validation details and any UI notes.
5. Prefer squash merges so `main` stays readable.

## Changesets And Versioning

We use [Changesets](https://github.com/changesets/changesets) to manage version bumps for the published package.

Add a changeset when a PR changes the shipped library in a way that affects consumers.

```bash
npm run changeset
```

Choose:

- `patch` for fixes and polish
- `minor` for backward-compatible additions
- `major` for breaking changes

You usually do not need a changeset for:

- CI or release workflow updates
- documentation-only changes
- demo-only changes that do not affect the published package

## CI Expectations

Every PR should pass:

- `npm run typecheck`
- `npm test`
- `npm run build`

GitHub Actions runs the same checks on pull requests and on `main`.

## Recommended GitHub Settings

Set branch protection on `main` to keep the repo flow clean:

- require the `CI / validate` check before merging
- require branches to be up to date before merge
- prefer squash merging
- restrict direct pushes to `main`

## Release Flow

1. Feature PRs merge into `main` with their changesets.
2. The release workflow updates or creates a release PR on top of `main`.
3. Merging that release PR updates `package.json`, creates the release tag, publishes to npm, and creates a GitHub release.

For this repo, the expected publish secret is:

- `NPM_TOKEN`

## Tagging

Release tags are generated by the release workflow from the published package version. Do not create manual version tags for normal releases unless you are intentionally handling an exceptional recovery.
